# Q Objects Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add Django-style `Q` objects so callers can compose filters with `&`, `|`, and `~`. Builds on the cross-class lookups (#40), lookup suffixes (#43), and reverse lookups (#46) ŌĆö every `(key, value)` leaf inside a Q tree uses the same key resolution and SPARQL emission as `RDFManager.filter` does today. Closes GitHub issue #48.

**Architecture:** A new `djangordf.query.Q` class holds three pieces: a connector (`AND` or `OR`), a negation flag, and a list of children. Each child is either another `Q` (subtree) or a `(key, value)` leaf. `Q(**kwargs)` builds a leaf-only AND node from the kwargs; `Q(*qs, _connector=...)` combines existing Q nodes. The operator dunders (`__or__`, `__and__`, `__invert__`) produce new Q trees without mutating their operands. `RDFManager.filter` becomes the thin shell that wraps positional `Q` args plus kwargs into one top-level AND Q, hands it to `RDFQuerySet`, and returns the queryset. The queryset's SPARQL builder walks the Q tree recursively: an AND node concatenates its children's patterns inside one group; an OR node emits each child in its own group joined by `UNION`; a NOT node wraps the child in `FILTER NOT EXISTS`. Variable counters thread through the walk so OR-children mint fresh names without clashing. Leaves dispatch to the existing path-walking helper (`_resolve_segment`, suffix builder, reverse handling) so every feature added in #40/#43/#46 keeps working inside Q.

**Tech stack:** Python 3.10+, rdflib 7.x, Django 3.2+, pytest 9.x, pytest-django 4.x.

**Spec reference:** [docs/superpowers/specs/2026-05-22-rdfmodel-walking-skeleton-design.md](../specs/2026-05-22-rdfmodel-walking-skeleton-design.md) ┬¦10. PR #40 and PR #44 both flagged `Q` composition as a future ticket.

**Issue:** [#48 Q objects with AND, OR, NOT for RDFManager.filter](https://github.com/judaicalink/djangordf/issues/48).

## Tasks

### Task 1 ŌĆö `Q` class

- [ ] `djangordf/query.py` ships:
      - `class Q` with class attributes `AND = "AND"`, `OR = "OR"`.
      - `__init__(self, *args, **kwargs)`: `_connector` and `_negated` are keyword-only (underscore-prefixed) so they cannot collide with filter kwargs; `args` are `Q` subtrees; `kwargs` become `(key, value)` leaves. Empty (`Q()`) raises `ValueError`.
      - `children: list` mixes Q nodes and `(key, value)` leaf tuples.
      - `__or__`, `__and__`: return a new Q with connector set accordingly; flatten same-connector children into one node.
      - `__invert__`: return a new Q copy with `negated` flipped.
      - `__bool__`: raise `TypeError` (mirrors Django; avoids accidental coercion).
      - `__repr__`: stable, useful for debugging.

### Task 2 ŌĆö Recursive SPARQL builder

- [ ] New private helper `_q_to_sparql(q, ctx)` returns a SPARQL fragment string. `ctx` carries:
      - `manager` ŌĆö for predicate resolution and `_object_term`.
      - `current_cls` ŌĆö the class the parent path landed on.
      - `current_var` ŌĆö the SPARQL variable used as the "subject" of this group.
      - `var_counter` ŌĆö shared counter (`list`/`box` so increments propagate).
- [ ] Leaf `(key, value)` ŌåÆ reuse the existing path-walker logic (extract it into a method that returns triple strings + filter strings rather than appending to lists). Wrap the result in `{ ... }` if needed by the caller.
- [ ] AND node ŌåÆ concatenate child fragments inside the same group; emit a flat `{ ŌĆ” }`.
- [ ] OR node ŌåÆ emit `{ child1 } UNION { child2 } UNION ...`.
- [ ] NOT node ŌåÆ wrap `{ ŌĆ” }` in `FILTER NOT EXISTS { ŌĆ” }`. Negated NOT-EXISTS does not bind variables to the outer scope, which is the desired semantics.
- [ ] Compose: a node that is negated AND has connector OR/AND wraps its body inside `{ ... }` first, then negates the wrapper with NOT EXISTS.
